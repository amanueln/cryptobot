# Dashboard TOTP Authentication — Design Spec

## Goal

Protect the dashboard with Google Authenticator (TOTP) so only the bot owner can access it. Auth is toggleable via env var so it doesn't interfere with local development.

## Problem

The dashboard has no authentication. Anyone who finds the URL can view positions, trigger scans, reset data, and (once live trading is wired up) execute trades. This is a security risk, especially on a network-accessible ZimaOS deployment.

## Design

### Toggle

`AUTH_ENABLED=true` in `.env` enables auth. If missing, empty, or `false`, the dashboard works exactly as it does now — no login, no middleware, no behavior change.

### Setup Flow (First Time)

When auth is enabled but no TOTP secret exists yet:

1. All routes redirect to `/setup`
2. Setup page generates a TOTP secret and displays:
   - A QR code (scannable by Google Authenticator)
   - The raw secret as text (for manual entry / copy-paste)
3. User scans or enters the secret in Google Authenticator
4. User enters the current 6-digit code to verify setup works
5. Backend saves the secret to a file (`data/.totp_secret`) and logs the user in
6. File is added to `.gitignore`

The secret is stored in `data/.totp_secret` (not `.env`) because:
- It's generated by the app, not configured by the user
- Avoids needing to parse/write `.env` files programmatically
- Simpler to check existence (`os.path.exists`) for setup detection
- Still gitignored and local to the machine

### Login Flow (Ongoing)

1. User opens dashboard → `@app.before_request` middleware checks for valid session cookie
2. Valid cookie → request proceeds normally
3. No cookie or expired → return 401 for API routes, redirect to `/login` for page routes
4. Login page shows:
   - Text input field for the 6-digit code (type/paste)
   - Submit button
5. User enters code from Google Authenticator
6. `POST /api/auth/verify` checks the code via `pyotp`
7. Valid → sets signed session cookie (30-day expiry) → redirects to dashboard
8. Invalid → shows error, lets user retry

### Session Management

- **Cookie name:** `bot_session`
- **Expiry:** 30 days from login
- **Signing:** Flask's `SECRET_KEY` (generated once, stored in `data/.flask_secret`)
- **Per-device:** Each browser/device gets its own independent session
- **Logout:** `POST /api/auth/logout` clears the cookie (accessible from a menu/button in the UI)

### Backend Changes (`dashboard/api/app.py`)

**New middleware — `@app.before_request`:**
- If `AUTH_ENABLED` is not `true`, skip entirely (no-op)
- If request path is in whitelist, skip: `/login`, `/setup`, `/api/auth/*`, static assets (`.js`, `.css`, `.ico`, etc.)
- Check for valid `bot_session` cookie
- If invalid/missing: return 401 for `/api/*` routes, redirect to `/login` for page routes

**New endpoints:**
- `POST /api/auth/verify` — accepts `{"code": "123456"}`, verifies against stored TOTP secret, sets session cookie
- `POST /api/auth/logout` — clears session cookie
- `GET /api/auth/setup` — generates new TOTP secret, returns `{"secret": "...", "qr_url": "otpauth://..."}` (only if no secret exists yet)
- `POST /api/auth/setup/confirm` — accepts `{"code": "123456"}`, verifies code against pending secret, saves secret to disk if valid

**Dependencies:**
- `pyotp` — TOTP generation/verification (standard, lightweight, no external calls)
- `qrcode` — QR code generation for setup page (renders as base64 PNG)

### Frontend Changes

**New: Login component** (`dashboard/ui/src/app/components/login/`)
- Centered card with app name/logo
- 6-digit code input (auto-focus, numeric)
- Submit button
- Error message on invalid code
- Standalone Angular component, outside the main app layout

**New: Setup component** (`dashboard/ui/src/app/components/setup/`)
- Centered card explaining what to do
- QR code image (from backend base64)
- Raw secret text with copy button
- Verification input + confirm button
- Success → redirects to dashboard

**ApiService changes:**
- HTTP interceptor catches 401 responses
- On 401 → redirects to `/login`
- Adds `withCredentials: true` to requests so cookies are sent

**Routing:**
- `/login` → LoginComponent
- `/setup` → SetupComponent
- All existing routes unchanged

**Logout button:**
- Small "Logout" link in the header bar
- Calls `POST /api/auth/logout` → redirects to `/login`
- Only shown when auth is enabled

### What's NOT in scope

- Multiple users / user accounts — single user only
- Password fallback — TOTP only
- Rate limiting on login attempts — can add later if needed
- Forced auth when live trading is enabled — future enhancement

## Files Changed

### Backend:
1. `dashboard/api/app.py` — auth middleware, 4 new endpoints, secret file management
2. `requirements.txt` or equivalent — add `pyotp`, `qrcode[pil]`
3. `.gitignore` — add `data/.totp_secret`, `data/.flask_secret`

### Frontend:
1. `dashboard/ui/src/app/components/login/login.component.ts` — new login page
2. `dashboard/ui/src/app/components/setup/setup.component.ts` — new setup page
3. `dashboard/ui/src/app/services/api.service.ts` — 401 interceptor, auth API calls
4. `dashboard/ui/src/app/app.routes.ts` — add `/login` and `/setup` routes
5. `dashboard/ui/src/app/app.component.ts` — logout button in header (conditional)
